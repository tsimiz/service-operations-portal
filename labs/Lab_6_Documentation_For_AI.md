Lab 6 — Documentation for AI
Generate documentation from the work, then review it

Time: 15:00–15:15
Audience: Everyone. Reviewing generated documentation is a skill non-technical participants take full part in.

Purpose

In the morning, the Doc agent updated the changelog and the documentation from the work that was just built. This lab is you being the Doc agent: generate operational and developer documentation from the feature you implemented in Lab 5, structured so that an agent could read it later.

Documentation is generated from the work, not written afterwards as a chore. The shift is in where the human effort goes: from writing the documentation to reviewing it and improving the pipeline and instructions that produced it.

Learning objectives

By the end of this lab, you can:

    generate documentation from an implemented feature rather than writing it from scratch
    update a changelog in a consistent, machine-readable format
    structure documentation for AI retrieval (headings, IDs, consistent shape)
    audit generated documentation as a deliverable, not accept it blindly
    optionally delegate a documentation run to a background agent and review the result

What you will use and produce

Read:

    your implemented code under src/main/
    docs/specs/spec-lab.md  (the feature's spec)
    docs/changelog.md  (the changelog; create it if it does not exist)

Produce:

    an updated docs/changelog.md entry for your feature
    any documentation file the change touched, updated
    optionally docs/operational/<feature>.md, generated by a background agent

Required files

    docs/specs/spec-lab.md  (the spec)
    .github/copilot-instructions.md  (conventions, applied automatically)

Important training design

Generated documentation is a deliverable like any other: you audit it, then you improve the pipeline and the instructions that produced it. The review is the human approval gate, and that gate is what seeds the Lab 7 canvas. Do not skip the review.

Exercise 6.1 — Update the changelog

Time: 15:00–15:05
Step 6.1.1 — Generate the entry

In Agent mode, prompt:

    Update docs/changelog.md with an entry for what I just implemented, including the REG-01 extension. Use today's date and a consistent format: a dated heading, then Added or Changed bullets, one line per change, referencing the class or endpoint.

Step 6.1.2 — Check it

Read the entry. Is it accurate? Does it match what you actually built? Fix anything wrong before continuing. A changelog the agent wrote is a draft you own.

Validation

[ ] docs/changelog.md has a dated entry for your feature
[ ] The entry mentions the REG-01 extension
[ ] The format is consistent (dated heading, bulleted changes)
[ ] The entry matches what you actually built

Exercise 6.2 — Update any documentation the change touched

Time: 15:05–15:09

Ask Copilot to check whether any other file under docs/ describes the feature area and is now out of date:

    Check whether any file under docs/ describes the service notes feature and is now stale given my change. If so, update it. Do not create new files unless the change clearly needs one.

Review and approve. Do not let it invent a pile of new documents; the point is keeping what exists accurate.

Validation

[ ] Any existing doc that described the feature is now accurate
[ ] No unnecessary new documentation files were created
[ ] You reviewed each change before approving

Exercise 6.3 — Structure for retrieval

Time: 15:09–15:12

Look at what was generated through the lens of an agent reading it later. Good structure means: clear headings, stable identifiers (the spec ID, the REG id), one topic per section, and a consistent shape. Adjust one thing to make the documentation easier for an agent to navigate, and say why.

This is the concrete version of the morning point: agents read structure, not prose, and well-structured docs cost fewer tokens to read, so quality and cost pull the same way.

Validation

[ ] The documentation uses clear headings and stable identifiers
[ ] You improved one structural thing and can say why it helps retrieval

Exercise 6.4 — Optional: delegate to a background agent

Time: 15:12–15:15 (only if your pair is ahead)

Hand a documentation run to a background agent and keep working while it runs, then review the result.

Step 6.4.1 — Delegate

Prompt a background agent:

    Generate an operational documentation section for the service notes feature, structured for AI retrieval, based on docs/specs/spec-lab.md and the changelog entry I just wrote. Write it to docs/operational/service-notes.md. Do not change application code.

Keep it on Default Approvals. Prefer a local background session over a cloud agent (no pull-request sprawl with a big room).

Step 6.4.2 — Review the result

When it finishes, review the output together: is it accurate, is it structured, what would you improve? That review is the human approval gate. It is exactly the moment the Lab 7 canvas is about: where does a human approve what an agent produced?

Validation

[ ] (Optional) A background agent produced docs/operational/service-notes.md
[ ] You reviewed it together and noted what you would improve
[ ] No application code was changed by the documentation run

Done when

You have:

[ ] A dated, consistent changelog entry covering the feature and the REG-01 extension
[ ] Any existing documentation the change touched, updated and accurate
[ ] Documentation structured for retrieval (headings, stable identifiers)
[ ] Reviewed every generated change as a deliverable you own
[ ] (Optional) Delegated a documentation run to a background agent and reviewed it
